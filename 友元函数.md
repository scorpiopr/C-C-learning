友元函数（Friend Function） 是一个定义在类外部的普通函数，但它需要在类内部进行声明，声明后它就拥有了访问该类所有 private（私有）和 protected（保护）成员的特权。
# 一、 友元函数的语法实现
要将一个函数变成友元函数，只需在类的内部对该函数进行声明，并在声明前加上关键字 friend。

```
#include <iostream>
#include <string>
class BankAccount {
private:
    std::string owner;
    double balance; // 私有成员，外部无法直接访问
public:
    BankAccount(std::string name, double money) : owner(name), balance(money) {}

    // 🔑 在类内部声明友元函数
    friend void auditAccount(const BankAccount& account);
};
// 外部普通的全局函数（注意：定义时【不需要】加 friend 关键字，也【不需要】加 BankAccount:: 作用域）
void auditAccount(const BankAccount& account) {
    // ✅ 因为是友元，所以可以直接访问 private 成员 balance 和 owner
    std::cout << "审核账目: " << account.owner 
              << " 的余额为 $" << account.balance << std::endl;
}
int main() {
    BankAccount my_bike("Alice", 5000.0);
    auditAccount(my_bike); // 像普通全局函数一样直接调用
    return 0;
}
```

# 二、 友元函数的核心特性与规则（面试高频）
它不是类的成员函数：
友元函数本质上依然是一个普通的全局函数，它没有 this 指针。
因此，在调用它时，不能通过对象去点出来（不能写成 obj.friend_func()），必须像普通函数一样直接调用（如 friend_func(obj)）。
声明位置无所谓：
它可以声明在类的 private、protected 或 public 任何区域，其访问权限和效果完全相同。
不能被继承：
“爸爸的朋友不一定是儿子的朋友”。基类的友元函数不会自动变成派生类的友元函数。
单向性（不对等性）：
“我把你当朋友，你不一定把我当朋友”。A 是 B 的友元，并不代表 B 是 A 的友元。
# 三、 为什么需要友元？它的核心应用场景
既然它破坏了面向对象的“封装性”，为什么 C++ 还要保留它？工程上它有两个不可替代的场景：

1. 运算符重载（Operator Overloading）—— 最核心的应用
当我们想重载流输入输出运算符（<< 和 >>）时，必须使用友元函数。
因为如果写成成员函数，左操作数必须是该类的对象（调用会变成 obj << std::cout，这不符合输入输出习惯）。为了让 std::cout 站在左边，必须写成全局函数，而全局函数又需要读写类的私有成员，此时友元是完美的解法：

```
class Point {
private:
    int x, y;public:
    Point(int x, int y) : x(x), y(y) {}

    // 重载 << 运算符，必须用友元
    friend std::ostream& operator<<(std::ostream& os, const Point& p) {
        os << "(" << p.x << ", " << p.y << ")";
        return os;
    }
};
```

2. 提高并发/底层框架的运行效率
在编写底层的矩阵运算库、图形学引擎或多线程通信框架时，如果频繁通过 get() 和 set() 函数去读写私有变量，会产生大量的函数调用开销。使用友元函数可以直接进行内存级别的读写，换取极致的性能。
