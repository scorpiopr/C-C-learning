
## 一、 析构函数无法被重载
## 1. 析构函数禁止带有任何参数
重载的底层前提是“函数签名（Function Signature）不同”，即函数名相同但参数的个数或类型必须不同。
然而，C++ 标准极其严格地规定：析构函数不能带有任何参数，甚至连 void 都不能写，也绝对没有返回值。
```
class Tester {public:
    ~Tester();       // ✅ 唯一合法的普通/虚析构函数
    ~Tester(int x);  // ❌ 编译直接暴毙！析构函数禁止带参，导致重载无从谈起
};
```
因为大家都不能带参数，它们的函数签名在编译期是完全一模一样的，自然无法并存。
## 2. 编译期决议的逻辑悖论：调用时无法进行“重载决议”
当一个对象生命周期结束（比如离开大括号作用域，或者执行 delete ptr;）时，系统会自动隐式调用析构函数。

* 痛点：因为析构函数是由编译器在底层自发、隐式调用的，它在调用时根本无法像普通函数那样，通过外部传入的实参（如 obj.func(10)）来让编译器在编译期进行「重载决议（Overload Resolution）」。
* 如果允许重载，当对象销毁时，编译器将完全不知道该去调用哪一个析构函数，逻辑直接锁死。

## 二、 析构函数可以被“重写”
在实现继承时，为保证能释放子类特有的成员，基类析构函数定义为虚函数，派生类重写析构函数
* 底层机制：
通常我们说重写，要求子类函数名和父类完全一致。但基类叫 ~Base()，子类叫 ~Derived()，名字根本不同，为什么能叫重写？
因为在 C++ 编译器底层，只要基类的析构函数加了 virtual 关键字，编译器就会在虚函数表（vtable）中为析构函数预留一个固定的槽位（通常在底层被统一命名为特殊的 deleting destructor）。
子类的析构函数虽然名字不同，但它会无情地覆盖（Override）掉虚表中基类析构函数的槽位！
```
class Base {public:
    virtual ~Base() { cout << "~Base"; } // 🔑 声明为虚析构
};
class Derived : public Base {public:
    ~Derived() override { cout << "~Derived"; } // 🔑 隐式重写！覆盖虚表槽位
};
```
正因为这种底层的虚表重写机制，当你使用基类指针 Base* ptr = new Derived(); delete ptr; 时，程序才能在运行期通过虚表，先精准调用子类的析构函数，再自动回溯调用父类的析构函数，完美防御线上内存泄漏灾难。
