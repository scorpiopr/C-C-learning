**宏定义只在预处理阶段进行自上而下的文本替换**，不具备任何运行期的作用域或生命周期概念。 宏的生效范围完全取决于它在源文件中的物理行顺序。

## 代码举例
```cpp
#include <stdio.h>
#define a 10 
 
void foo();  
int main(){ 
   printf("%d..", a); 
   foo(); 
   printf("%d", a); 
} 
void foo(){ 
   #undef a 
   #define a 50 
}
```
## ⚙️ 运行结果
```
10..10
```
## 预处理器的文本替换轨迹（从上到下）
当预处理器自上而下扫描这段代码时，它的视角是这样的：

* 第 2 行：遇到 #define a 10。预处理器记下：从现在开始，后面只要见到 a，就替换成 10。
* 第 6 行（main 函数内部）：遇到 printf("%d..", a);。因为此时 a 还是 10，所以被替换为 printf("%d..", 10);。
* 第 7 行：遇到 foo();。这是一句普通的运行期函数调用，预处理器直接跳过。
* 第 8 行：遇到 printf("%d", a);。此时宏 a 依然是 10，所以被替换为 printf("%d", 10);。
* 第 11 行（foo 函数内部）：遇到 #undef a。预处理器取消之前 a = 10 的定义。
* 第 12 行：遇到 #define a 50。预处理器记下：从这一行往后，如果再出现 a，就替换成 50。但可惜的是，这一行后面已经没有任何代码使用 a 了。

## 最终编译时（运行前）的代码全貌
经过预处理后，送入真正编译器的 .i 文件（临时文件）中，main 函数其实长这样：
```cpp
int main(){ 
   printf("%d..", 10); // a 已经被换成了 10
   foo(); 
   printf("%d", 10);   // a 已经被换成了 10
}
```
此时 foo() 函数内部已经变空了。程序运行时，无论 foo() 被调用多少次，main 函数里的两个 10 都是写死在机器码里的，绝对不会改变。
