# 以下 7 种典型场景绝对无法进行函数重载：
------------------------------
## 一、 场景 1：仅有返回值类型（Return Type）不同
这是全行业出镜率最高、最经典的面试陷阱题。
```
int func(int x);
void func(int x); // ❌ 编译报错：无法仅靠返回值类型进行重载！
```

* 底层原理：编译器在进行重载决议时，主要依据的是函数的调用上下文。如果你直接写一句 func(10);（忽略了返回值），编译器翻遍参数列表发现它们完全一样，根本无法通过调用者来断定到底该调用哪一个，因此在语法层面直接封死。

------------------------------
## 二、 场景 2：仅有 static（静态成员函数）修饰符的差异
在一个类内部，普通成员函数和静态成员函数无法仅靠一个 static 关键字并存。
```
class Tester {public:
    void func(int x);
    static void func(int x); // ❌ 编译报错：普通成员与静态成员无法这样重载
};
```

* 底层原理：普通成员函数在编译后，其参数列表的头部会被编译器秘密塞入一个隐式的 this 指针（实际形参退化为 Tester* const this, int x） [🌐]；而 static 函数没有 this 指针 [🌐]。然而，如果从类外部通过类名调用 Tester::func(10)，或者在某些不涉及实例的上下文中，编译器依然会产生致命的调用歧义。

------------------------------
## 三、 场景 3：参数列表仅有顶层 const（Top-level const）的差异
这是一个非常隐蔽的指针/常规变量陷阱。
```
void print(int x);
void print(const int x); // ❌ 编译报错：顶层 const 无法构成重载！
void ptr_func(int* p);
void ptr_func(int* const p); // ❌ 编译报错：指针的顶层 const 同样无法构成重载！
```

* 底层原理：顶层 const 意味着变量本身是只读的。对于传值调用的函数，形参本身只是外部实参的一份局部临时拷贝。在函数内部它是不是 const，对外部调用者（如 print(a)）没有任何影响。因此，它们在编译期生成的修饰名（Mangled Name）完全一致。
* 💡 绝杀对比（底层可重载）：如果是底层 const（Low-level const，修饰指针所指向的对象或引用），则可以完美重载！
```
void print(int& x);
void print(const int& x); // ✅ 完美重载！一个是左值引用，一个是常量左值引用（可绑定右值）。
```

------------------------------
## 四、 场景 4：参数仅有“数组名”与“普通指针”的类型伪装
我们在复习第一课时学到的「数组名退化机制」，在这里会直接扼杀重载。
```
void process(int* arr);
void process(int arr[]);   // ❌ 编译报错：数组形参本质就是指针void process(int arr[10]); // ❌ 编译报错：即使指定了长度，底层依然是 int*
```

* 底层原理：在 C++ 中，内置数组作为函数传值参数时，在编译期会无脑且自动地退化为指向首元素的普通指针（T[] 彻底退化为 T*）。这三个函数在编译器眼里，它们的函数签名全部是 void process(int*)，属于同一函数的重复声明。

------------------------------
## 五、 场景 5：形参仅有“别名（typedef / using）”的差异
很多开发者喜欢为了可读性定义大量的别名，但别名并不能创造新的类型。
```
using Identifier = int;
typedef int RealID;
void check(Identifier id);
void check(RealID id);    // ❌ 编译报错：本质上都是 int，无法重载！
```

* 底层原理：typedef 和 using 在现代 C++ 中仅仅是类型别名（Type Alias），在编译期展开后，它们会无缝还原为其背后的原生基础类型。因此对编译器而言，参数列表没有任何改变。

------------------------------
## 六、 场景 6：析构函数（Destructor）
这是上一轮我们刚刚完美荡平的硬核语法死穴。
```
class Dummy {public:
    ~Dummy();
    ~Dummy(int x); // ❌ 编译直接暴毙！析构函数禁止带任何参数，无法形成签名差异。
};
```

* 底层原理：析构函数由编译器在对象生命周期终点自发、隐式调用，语法强制要求禁止携带任何参数，因而天然丧失了重载的前提。

------------------------------
## 七、 场景 7：函数被包裹在 extern "C" 块中（跨语言调用限制）
当你需要编写 C/C++ 混合项目，将函数暴露给纯 C 语言环境或动态链接库（DLL）时：
```
extern "C" {
    void display(int x);
    void display(double x); // ❌ 编译报错：在 extern "C" 作用域内禁止重载！
}
```

* 底层原理：C 语言是完全不支持函数重载的。为了让 C 语言能够调用 C++ 编译出来的函数，extern "C" 会强行命令 C++ 编译器关闭名修饰（Name Mangling）机制。
* 在该作用域内，display(int) 编译出来的底层汇编符号名就是极其朴素的 _display（而不是 C++ 带有类型后缀的魔改名）。如果允许重载，两个函数都会生成相同的 _display 符号，导致链接器（Linker）当场发生「符号重定义（Symbol Redefinition）」而崩盘。
